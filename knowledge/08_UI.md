# Knowledge: UI

> **Domain:** Presentation layer, user experience, interaction patterns, information display
> **Audience:** AI agents implementing or modifying UI components
> **Version:** 1.0

---

## 1 — Purpose

The UI is the window through which the player experiences the simulation. Every fighter's rise, every financial crisis, every championship victory — all of it reaches the player through the UI. The UI does not create these experiences; it presents them. It translates raw simulation state into something the player can understand, interact with, and care about.

The UI is not the game. The game is the simulation — the fighters, the economy, the world. The UI is the delivery mechanism. A beautiful UI on a broken simulation is a polished lie. A functional simulation with a confusing UI is a hidden gem. The goal is both: a deep simulation delivered through an interface that makes the player feel in control.

---

## 2 — UI Philosophy

### The UI is a window, not the room

The player looks through the UI to see the simulation, but the simulation exists independently. If the UI were removed, the world would continue ticking, fighters would still train, and titles would still change hands. The UI adds visibility and interactivity — it does not add existence.

### State flows one way

Simulation state flows into the UI. User actions flow out of the UI. There is no two-way binding, no UI-owned state that the simulation depends on. The UI reads state, renders it, and dispatches player intentions. The simulation processes those intentions and produces new state. The cycle repeats.

### The UI should make the player feel capable

A good UI makes the player feel like a competent manager. Information is where they expect it. Actions produce clear results. Mistakes are recoverable. A bad UI makes the player feel lost — information is buried, buttons do unexpected things, and the player blames themselves for the interface's failures.

### Clarity over cleverness

A UI that is obvious on first glance is better than one that is elegant but confusing. The player should never have to "figure out" how to do something basic. Novel interactions should be reserved for novel features — routine actions should follow routine patterns.

---

## 3 — Design Goals

- **Information is findable** — the player can locate any piece of information within seconds.
- **Actions are predictable** — clicking a button does what the player expects, every time.
- **Feedback is immediate** — every interaction produces a visible response within 100ms.
- **State is visible** — the player can see the current state of their camp without clicking through multiple screens.
- **Complexity is revealed progressively** — new players see simple views; advanced players discover depth.
- **Errors are recoverable** — mistakes can be undone, dangerous actions ask for confirmation.
- **The UI is consistent** — similar things look similar, similar actions work similarly.

---

## 4 — Design Principles

| Principle | Meaning |
|-----------|---------|
| **UI Reflects State** | Every visible element corresponds to a value in the simulation state. No UI element exists without a state backing it. |
| **Single Source of Truth** | The simulation state (`g`) is the only source of what is displayed. The UI never maintains its own copy of game data. |
| **Stateless Where Possible** | UI components should derive their display from props, not from internal state. Internal state is for UI concerns only (open/closed, active tab, scroll position). |
| **Predictable Interactions** | The same action in the same context always produces the same result. No hidden modes, no context-sensitive behavior that isn't obvious. |
| **Consistent Patterns** | If a fighter card looks one way on the Roster screen, it should look the same way on the Rankings screen. Consistency builds recognition. |
| **User Intent First** | The UI should interpret what the player means, not just what they click. A fighter card click means "show me more about this fighter," regardless of where it appears. |
| **Immediate Feedback** | Every button click, every keypress, every interaction produces a visible change within one frame. Silence = broken. |
| **Presentation Over Computation** | The UI formats, colors, and arranges data. It does not calculate, derive, or transform it beyond display formatting. |

---

## 5 — Non Goals

- **Not where business logic lives** — calculations, validations, and rules belong in the engine, not in UI components.
- **Not a state management system** — the UI reads from the simulation state; it does not maintain parallel state.
- **Not a data transformation layer** — formatting (rounding numbers, adding currency symbols) is acceptable. Transforming data (recalculating rankings, deriving new values) is not.
- **Not responsible for game balance** — the UI presents what the simulation produces. If the numbers are wrong, fix the simulation, not the display.
- **Not an animation engine** — subtle transitions enhance feel, but the UI is not a cinematic experience.
- **Not a tutorial system** — onboarding guides the player, but it lives outside the core UI components.

---

## 6 — Responsibilities

### UI owns

- Displaying simulation state to the player (rendering fighters, finances, rankings, events)
- Capturing player input (clicks, keyboard, form entries)
- Dispatching player intentions to the engine (via `dispatch()` or `up()`)
- Navigation between views (tabs, modals, screens)
- Visual feedback for player actions (highlights, transitions, notifications)
- Form validation for user input (before dispatching to engine)
- Confirmation dialogs for destructive actions
- Loading, empty, and error states for every view
- Accessibility (keyboard navigation, screen reader support, focus management)
- Responsive layout (adapting to different screen sizes)

### UI does NOT own

- Game logic — combat resolution, training calculation, economic settlement
- State mutation — the UI never modifies `g` directly
- Save/load logic — the UI triggers save/load but does not implement it
- Event generation — the UI displays events from the inbox, does not create them
- Data validation beyond input format — "is this a number?" is UI; "can this fighter afford this contract?" is engine
- Game balance — the UI renders whatever the simulation produces

### The UI is a Mirror, Not a Brain

AI should think of the UI as a mirror that reflects the simulation. If the reflection is wrong, fix the mirror (the UI rendering). If the thing being reflected is wrong, fix the simulation (the engine). Never fix simulation problems by changing the mirror.

## 8 — System Relationships

### Reads (UI observes)

| System | What UI Reads | How It's Displayed |
|--------|--------------|-------------------|
| **Fighter** | Name, attributes, record, morale, contract, training, injury | Fighter cards, detail views, roster tables |
| **Coach** | Name, skill, salary, specialty | Coach cards, staff list |
| **Camp** | Tier, reputation, chemistry, facilities | Dashboard KPIs, facility view |
| **Economy** | Cash, sponsors, income, expenses | Finance tab, dashboard summary |
| **World** | Week counter, rankings, divisions, AI fighters | Rankings view, division browser |
| **Events** | Inbox messages, timeline entries | Inbox list, timeline view |
| **Achievements** | Unlocked achievements, progress | Achievement grid, notifications |
| **Settings** | Language, UI preferences | Applied globally |

### Triggers (UI sends to Engine)

| Player Action | UI Event | Engine Dispatch |
|---------------|----------|-----------------|
| Click "Train" button | Select training program | `dispatch({ type: "SET_TRAINING", ... })` |
| Press Space | Advance week | `advance()` → `tick(g)` |
| Click "Accept Fight" | Accept fight offer | `dispatch({ type: "ACCEPT_FIGHT", ... })` |
| Click "Hire Coach" | Hire a coach | `dispatch({ type: "HIRE_COACH", ... })` |
| Click fighter card | View fighter details | `setTab("fighterDetail")` |
| Type in form field | Input validation | Local state (until submit) |
| Submit form | Dispatch validated action | `dispatch({ type, payload })` |
| Navigate tab | Switch view | `setTab(tabName)` |

### Never Owns

The UI never owns: combat logic, economy logic, training logic, save logic, simulation logic, event generation, ranking calculation, or any business rule. The UI asks; the engine decides.

### State Change Propagation

When the simulation changes state, the UI must reflect it. This is the propagation path:

```
Tick completes → g updated
                    │
                    ▼
              React re-renders
                    │
        ┌───────────┼───────────┐
        ▼           ▼           ▼
    Dashboard    Roster      Finance
    refreshes    refreshes   refreshes
        │           │           │
        ▼           ▼           ▼
    Cash shows  Fighter      Sponsor
    new total   shows new    shows new
                training     income
                progress
```

Every component that reads from `g` re-renders with the new state. No component needs to be "notified" individually — they all read from the same source and React handles the update propagation.

## 11 — Business Rules

### Form Validation

- Validate input format at the UI level before dispatching to the engine.
- Numeric fields: is this actually a number? In range?
- Required fields: is this filled in?
- Invalid input shows an inline error message near the field.
- Business rule validation (can the player afford this? is this fighter eligible?) happens in the engine, not the UI.
- The engine may reject an action even after UI validation passes — the UI must handle rejection gracefully (show engine error message).

### Loading State

- When data is being fetched or computed, show a loading indicator.
- Loading indicators should be subtle — spinner, skeleton, or dimmed content.
- The game should remain interactive during loading where possible.
- Long operations (build, deploy) should show progress.

### Error State

- When something goes wrong, the UI explains what happened in plain language.
- Error messages are specific: "Cannot hire coach: not enough cash ($12,000 needed, $8,500 available)" not "Error."
- Errors are displayed near the relevant UI element, not in a generic popup.
- The player should always have a path forward — "Go back," "Try again," or "Fix the issue."

### Empty State

- When a list or view has no data, show a helpful message, not blank space.
- "No fighters booked. Visit the Inbox to find fight offers."
- "No sponsors yet. Build your reputation to attract sponsors."
- Empty states guide the player toward their next action.

### Confirmation

- Destructive or irreversible actions require confirmation.
- "Are you sure you want to fire Coach X? This cannot be undone."
- Confirmation dialogs are simple: action description + Confirm/Cancel.
- Non-destructive actions (changing training, viewing details) do not need confirmation.
- The player can disable confirmations for specific action types if they choose.

### Navigation

- Primary navigation (tabs) is always visible and consistent.
- The current tab is visually highlighted.
- Switching tabs preserves the player's context — no data is lost.
- Deep navigation (fighter detail from roster) has a clear back button.
- The player always knows where they are and how to get back.

### Modal Behavior

- Modals are used for focused interactions: negotiations, fight night, confirmations.
- Only one modal is visible at a time.
- Clicking outside a modal or pressing Escape closes it (if safe to do so).
- Modals trap focus — Tab cycles through modal elements only.
- The background is dimmed to indicate modality.

### Notifications

- Non-blocking notifications inform the player of background events.
- Notifications appear briefly and auto-dismiss.
- Important notifications (contract expiry, bankruptcy warning) persist until dismissed.
- The player can access notification history through the inbox.

### Undo

- The last action can be undone (Ctrl+Z or Undo button).
- Undo restores the exact state before the action.
- Redo (Ctrl+Y) restores an undone action.
- Not all actions are undoable — destructive actions with cascading effects may not support undo.
- The player is informed when an action cannot be undone.

## 12 — UI Philosophy & User Experience

### Don't surprise the player

The UI should behave consistently. If clicking a fighter card opens a detail view on one screen, it should do the same on every screen. Surprising behavior breaks trust and forces the player to learn the UI through trial and error.

### Reveal complexity progressively

New players should see the essentials: roster, inbox, advance week. Advanced features (detailed training philosophy, rivalry tracking, dynasty stats) should be discoverable but not overwhelming. The player uncovers depth as they explore, not as they are forced through tutorials.

### Important information first

The Dashboard should surface the most critical information: upcoming fights, financial health, injured fighters. Detailed breakdowns belong on dedicated screens. The player should be able to glance at the Dashboard and understand the state of their camp in seconds.

### Minimize unnecessary clicks

Common actions should be reachable in one click. If the player changes a fighter's training program 20 times per session, that action should not require navigating through three screens. Frequent actions deserve prominent placement.

### Fast feedback builds trust

When the player clicks a button, something should happen immediately — even if the full result takes time to process. A button press with no visible response feels broken. Disable the button, show a spinner, animate the transition — anything that says "I heard you."

### Consistent interaction patterns

If training selection uses a dropdown on the Roster screen, it should use a dropdown on the Fighter Detail screen. If the Finance tab uses tables for data, the Rankings tab should use tables too. Consistency makes the UI predictable; predictability makes it invisible.

### Respect player attention

Don't interrupt the player with modals, notifications, or animations while they're focused on a task. Batch non-urgent notifications. Defer animations that would block interaction. The player's attention is the most valuable resource in the UI.

### Accessibility is not optional

Every interactive element must be keyboard-accessible. Color must not be the only way to convey information (red text should also have an icon or label). Font sizes should be readable. The UI should work for players with different abilities.

## 13 — AI Decision Heuristics

When modifying or extending the UI, follow these heuristics.

### Don't put business logic in UI components

If you find yourself writing `if (fighter.record.w > 10)` inside a component to decide what color to show, that logic belongs in the engine or a utility function. The component should receive a pre-computed value or a simple flag. UI components render; they don't calculate.

### Don't duplicate domain validation

If the engine validates that a fighter must be uninjured to accept a fight, the UI should not duplicate that check. The UI can disable the button for injured fighters (convenience), but the engine is the authority. Duplicated validation creates divergence — the UI says "you can't" but the engine would have allowed it, or vice versa.

### Use existing UI patterns before creating new ones

Before building a new component, check if an existing component can be adapted. A new dashboard widget should follow the same card pattern as existing widgets. A new form should use the same validation style as existing forms. Consistency across the UI is more valuable than a slightly better design for one screen.

### Don't create new screens when existing ones suffice

A new feature doesn't automatically need a new screen. Can it be a section on an existing screen? A tab within an existing view? A modal? New screens add navigation complexity. Add them only when the feature genuinely needs dedicated space.

### Maintain visual and interaction consistency

If you change the color of "positive" from green to teal, change it everywhere. If you change how dropdowns work on one screen, change them on all screens. Partial consistency is worse than no consistency — it trains the player that the rules change depending on where they are.

### Test with real game state

Don't test UI changes with perfect, idealized data. Test with: a roster of 20 fighters, a fighter on a 10-fight losing streak, zero cash, maximum cash, all facilities at max level. Edge cases in data produce edge cases in rendering.

### Every interaction needs visible feedback

A button that silently dispatches an action with no visual change is indistinguishable from a broken button. Even if the state change is minor, show something — a brief highlight, a log entry, a number animation.

## 14 — Extension Strategy

### Adding a screen

1. Define the screen's purpose — what information does it show? what actions can the player take?
2. Design the layout — what goes where? follow existing screen patterns.
3. Build the component — read from `g`, dispatch actions.
4. Add navigation — tab in sidebar, route in App shell.
5. Test with diverse game states — early game, late game, edge cases.
6. Ensure the new screen feels consistent with existing screens.

### Adding a dialog or modal

1. Define the interaction — what does the player need to decide or input?
2. Use the existing modal pattern (overlay, focus trap, close behavior).
3. Keep the modal focused — one purpose, limited options.
4. Ensure keyboard accessibility — Escape closes, Tab cycles, Enter confirms.
5. Test: does the modal work when nested? (modal on top of another modal should not happen, but handle gracefully if it does).

### Adding a dashboard widget

1. Define the information — what KPI or summary does this widget show?
2. Keep it compact — dashboard widgets should be glanceable.
3. Make it interactive — clicking the widget navigates to the relevant detail screen.
4. Follow existing widget patterns (card layout, data formatting).
5. Test at different screen widths — widgets should reflow, not break.

### Adding a form

1. Define the fields — what does the player need to input?
2. Add input validation — format checks at the UI level.
3. Provide clear labels and placeholders.
4. Show validation errors inline, near the relevant field.
5. Add a submit button that dispatches the action.
6. Handle the engine response — success feedback or error display.
7. Follow existing form patterns (layout, styling, validation style).

### Adding a notification type

1. Define the trigger — what event produces this notification?
2. Determine the urgency — auto-dismiss or persist?
3. Design the notification appearance — follow existing notification patterns.
4. Add the notification to the notification system (not ad-hoc).
5. Ensure it doesn't interrupt the player unnecessarily.

## 15 — Golden Rules

These are the non-negotiable rules of the UI. Every UI decision must satisfy them.

| # | Rule | Why |
|---|------|-----|
| 1 | **UI displays state, never owns it** | The simulation is the truth. The UI is a mirror. If the mirror shows something wrong, fix the simulation, not the mirror. |
| 2 | **Keep business logic outside UI** | Business logic in the UI cannot be tested independently, cannot be reused, and creates hidden dependencies. Logic belongs in the engine. |
| 3 | **Reuse components before creating new ones** | Every new component is a new thing for the player to learn and for the developer to maintain. Exhaust existing patterns first. |
| 4 | **Consistency beats creativity** | A slightly ugly but perfectly consistent UI is better than a beautiful but inconsistent one. Players learn patterns, not individual screens. |
| 5 | **Fast feedback builds trust** | Every action produces a visible response. Silence is interpreted as failure. |
| 6 | **Every interaction has visible feedback** | If nothing changes when the player clicks, they will click again — or assume the game is broken. |
| 7 | **Important actions ask for confirmation** | Firing a coach, releasing a fighter, spending a large sum — these deserve a pause. Routine actions do not. |
| 8 | **Errors explain what happened and what to do** | "Something went wrong" is not an error message. "Cannot hire coach: $12,000 needed, $8,500 available" is. |

## 16 — Invariants

These must never be violated. If a change breaks one, the change is wrong.

- **UI never mutates game state directly** — all state changes go through `dispatch()` or `up()`. Direct mutation of `g` from a UI component is forbidden.
- **UI never imports from engine internals** — only public engine exports. Internal modules (individual resolvers, generators, config) are off-limits.
- **Every interactive element is keyboard-accessible** — Tab to focus, Enter/Space to activate, Escape to close.
- **Color is never the sole indicator of meaning** — red text for negative must also have an icon, label, or pattern.
- **The current tab is always visibly indicated** — the player never wonders "which screen am I on?"
- **No information is permanently hidden behind interactions** — tooltips and expandable sections are fine; essential data that requires clicking through 3 levels is not.
- **The UI works at 1024px width minimum** — not every player has an ultrawide monitor.
- **Empty states show guidance, not blank space** — every list and view handles the zero-case.
- **Loading states exist for any operation that takes more than 100ms** — the player should never stare at a frozen screen wondering if the game crashed.
- **The UI does not crash on unexpected state** — if the simulation produces something weird, the UI renders what it can and logs a warning. A broken render is better than a white screen.

## 17 — Common Mistakes

### Putting business logic in components

Writing `if (cash < 5000) showWarning = true` inside a Dashboard component instead of using a pre-computed flag or utility function. The component now owns a business rule that should live in the engine. When the rule changes, someone must remember to update the component.

### Duplicating validation

The UI checks "is the fighter injured?" and disables the fight button. The engine also checks "is the fighter injured?" and rejects the action. The UI check becomes stale when injury rules change. The engine check is the authority — the UI should trust the engine, not duplicate it.

### Creating new patterns for one-off cases

Adding a special card style for champion fighters that doesn't follow the existing card pattern. The player learns this exception but it doesn't generalize. Champion fighters should be distinguished within the existing card pattern (badge, border, icon), not with a completely different card.

### Hiding information behind unnecessary interactions

Putting fighter morale inside a tooltip on a detail screen that requires two clicks to reach. The player never sees morale and wonders why their fighters underperform. Important information should be visible at the point of decision-making.

### Inconsistent interaction patterns

A dropdown on the Roster screen vs a modal on the Fighter Detail screen for the same action (changing training program). The player learns two different patterns for the same task and hesitates each time.

### Over-notification

Every minor state change produces a notification. The player learns to ignore all notifications, including the important ones. Notifications should be reserved for events the player genuinely needs to know about.

### Ignoring empty states

A list that shows nothing when empty — no message, no guidance, just white space. The player wonders if the game is broken. Every list needs a zero-state message.

### Forgetting keyboard accessibility

Building a beautiful modal that can only be closed by clicking an X button. Keyboard-only players are trapped. Escape must close modals. Tab must cycle through interactive elements.
